# 実装計画: 画像の結合によるOCRのバッチ処理化（高速化）と解像度非依存化

ご指摘ありがとうございます！現在の実装では座標（`761, 986, 1067, 1806` など）が絶対値のピクセルでハードコーディングされているため、フルHD（1920x1080）以外の解像度の動画を読み込むとエリアがズレてしまいます。
ご提案の通り、割合（比率）ベースでのエリア計算を実装計画に追加します。

## 提案される変更内容

### 1. エリア指定の「割合（パーセンテージ）」計算の導入
現在ハードコーディングされているピクセル値を、元の想定解像度（1920x1080）を基準とした「割合」に変換します。動画を読み込んだ際、実際の動画の解像度（幅と高さ）にこの割合を掛け算することで、どんな解像度の動画でも常に正確なエリアを切り抜けるようにします。
- **全体領域の切り抜き**: 幅・高さに対するパーセンテージで動的計算。
- **4つの効果枠の境界線**: 切り抜いた領域（ROI）の高さに対するパーセンテージで動的計算。

### 2. 縦結合によるバッチ処理の導入（OCR呼び出しの1/4化）
テキストが切り替わった（検知された）フレームの画像を一時的にリストに貯めます。
バッファに **4枚** の画像が貯まったら、それらを縦に繋げて1枚の大きな画像（縦長）にします。

> [!TIP]
> なぜ「4枚」なのか？
> 解像度がフルHDの場合、抽出枠の高さは約 225px です。4枚縦に繋げると高さが 900px になります。
> PaddleOCRのデフォルト設定では「画像の一辺が960pxを超えると強制縮小する」という仕様があるため、画質を保ったまま処理できる安全な限界が「4枚」となります。

### 3. 結合画像からのデータ逆算ロジック
結合した画像に対して1回だけ `ocr.predict` を実行し、OCR結果のY座標を使って元の画像と枠に逆算して振り分けます。
この際の「225pxごとに区切る」という処理も、解像度に合わせて動的に計算された「ROI枠の実際の高さ」を使って計算するように実装します。
